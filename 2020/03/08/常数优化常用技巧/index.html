常数优化常见误区

Posted on 2020-03-08

常数优化是一个 OIer 必须要面对的一个问题，这个问题常常能将人们弄得焦头烂额，有时候用尽了常数优化的技巧也无法达到时间限制。随着 OI 的发展，常数优化慢慢成为了仅次于复杂度优化的必备技能，WC2015 也考察了常数优化的技巧。常数优化有很多常见的误区，让我们来一一看看吧：

前言

全文使用 GCC 8.1.0，环境为 64-bit Windows 10 Pro。

位运算优化

相信不少人在写读写优化的时候，都会写上这样一句奇怪的语句：

我认为这句话的 (ret << 1) + (ret << 3) 是完全没有必要的，可以直接用 ret * 10 代替。

为什么呢？首先，永远要记住一点，编译器不是傻子。编译器会在编译的过程中优化你的代码，无论你是否开启了 O2 等优化选项。

有的人会说，他测试过，(ret << 1) + (ret << 3) 的确能够加速代码。可是我们知道，代码的运行时长不仅仅和代码本身有关，不同的时间，代码的速度自然不同。举个例子，如果你的电脑什么都不开，只跑个 test.exe，速度自然是要比同时开 QQ、微信、VS、Chrome 的 test.exe 要快了。

那怎么判断代码是否更优呢？我们可以查看编译器产生的汇编代码，来看看是否真的进行了优化。

你会注意到，编译器并没有使用 imull 这样简单粗暴的乘法，而是使用左移运算来做这个乘法。你写的 (ret << 1) + (ret << 3)，编译器产生的汇编语言也和这个差不多（如果你的 test.asm 还保留着 a = a * 10 产生的汇编代码，编译器甚至都不会干任何事，因为它知道无需再编译一遍）。因此，完全没有必要将 foo * 10 写成 (foo << 1) + (foo << 3)。

综上所述，编译器并不会注意你代码中 foo * 10 和 (foo << 1) + (foo << 3) 的区别，所以考虑可读性的话，还是建议各位以后就写 foo * 10，位运算优化什么的交给编译器干就好啦~

inline

inline 已经被很多人滥用了，评测里、题解里，瞧去，一大片一大片满是的。有些函数，可能长达几十行，甚至还有递归……

我们不得不提出一个问题，inline 到底是有什么用呢？

首先，函数是怎么存储的？定义一个函数 foo()，实际上 foo 是一个指针，指向内存中 foo 函数的机器代码的开头。每当要调用这个函数了，就根据这个指针运行机器代码。

而有时候，我们想要书写一个非常简单的函数（比如平方），可是又不想写一个单独的函数（因为这样子时间开销较大），那该怎么办呢？

在 C 语言中，通常使用宏来解决这个问题。宏有一个优点：它在预处理器那里就将被替换好，而且还有 ##、# 等运算符，能比较方便地解决问题。有优点，自然也就有缺点，那就是传给宏的“参数”容易被重复计算，并且容易产生各种优先级之类的问题。举个例子：

C++ 引入了内联函数这个概念来解决这个问题。内联函数实质上并不是真正的函数，它在编译的时候就将被展开（就像宏那样），但是又不像宏那样容易出错。

在使用的时候务必要注意，inline 标识符只是用来_提醒_编译器可以把该函数变为内联函数，而非强制将该函数变为内联函数。如果需要强制，可以使用宏。

如果你进行了比较多次的实验，会发现——如果不加 inline，编译器就会先考虑作为普通函数，然后发现——普通函数的开销一点都不大，可以接受；如果加了 inline，编译器会先考虑作为内联函数。
